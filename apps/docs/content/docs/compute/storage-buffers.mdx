---
title: Storage Buffers
description: Read and write structured GPU data from shaders.
---

Storage buffers hold arbitrary structured data that shaders can read and write - unlike uniform buffers (small, read-only, fast) or vertex buffers (fixed layout for drawing).

They are the backbone of compute shaders and **GPU-driven rendering** (vertex pulling, particles, instancing).

## WGSL declaration

```wgsl
struct Particle {
  pos: vec2f,
  vel: vec2f,
}

@group(0) @binding(0) var<storage, read_write> particles: array<Particle>;
```

Access modes:

| Mode | Use |
| ---- | --- |
| `read` | Input data in compute or render |
| `read_write` | Compute output (same dispatch must not read what another thread writes hazardously) |
| `read` in render after compute write | Safe across separate passes |

## Creating a buffer

```tsx
const buffer = device.createBuffer({
  size: particleCount * 16, // bytes
  usage: GPUBufferUsage.STORAGE | GPUBufferUsage.COPY_DST,
});

device.queue.writeBuffer(buffer, 0, initialData);
```

For data consumed as vertex input **and** written by compute:

```tsx
usage: GPUBufferUsage.STORAGE | GPUBufferUsage.VERTEX | GPUBufferUsage.COPY_DST,
```

[Particles](/docs/compute/particles) uses this dual-purpose pattern.

## Vertex pulling

Instead of traditional vertex attributes, the vertex shader reads positions from a storage buffer:

```wgsl
@vertex
fn vs_main(@builtin(vertex_index) vi: u32) -> @builtin(position) vec4f {
  let circle = circles[vi / 6]; // 6 vertices per circle
  // compute corner from vi % 6…
}
```

This lets compute shaders animate vertex data without CPU readback - useful for procedural geometry and simulations.

## CPU readback

To read results on CPU (picking, debugging, ML postprocessing):

```tsx
const readBuffer = device.createBuffer({
  size,
  usage: GPUBufferUsage.MAP_READ | GPUBufferUsage.COPY_DST,
});
encoder.copyBufferToBuffer(gpuBuffer, 0, readBuffer, 0, size);
await readBuffer.mapAsync(GPUMapMode.READ);
const data = new Float32Array(readBuffer.getMappedRange());
readBuffer.unmap();
```

Readback is slow - prefer keeping simulation on GPU and only read when necessary. For worklet-thread readback, see [Reanimated](/docs/integrations/reanimated).
