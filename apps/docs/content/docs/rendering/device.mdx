WebGPU rendering starts with a **device** - your handle for creating buffers, textures, pipelines, and command encoders. React Native WebGPU exposes the same flow as the browser:

```tsx
const adapter = await navigator.gpu.requestAdapter();
const device = await adapter!.requestDevice();
```

The adapter represents a physical or virtual GPU. The device is the object you use for all subsequent GPU work.

## useDevice

[`useDevice`](/api/use-device) requests an adapter and device when a `Canvas` mounts - convenient for single-scene apps:

```tsx
import { useDevice, warnIfNotHardwareAccelerated } from "react-native-webgpu";

function Scene() {
  const { adapter, device } = useDevice();

  useEffect(() => {
    if (adapter) warnIfNotHardwareAccelerated(adapter);
  }, [adapter]);

  if (!device) return null;
  // configure context, build pipelines…
}
```

<Callout type="warn" title="Emulators">
Android emulators often use a **software fallback adapter** - rendering is slow and some features may be unavailable. `warnIfNotHardwareAccelerated` checks `adapter.info.isFallbackAdapter` and logs a warning. Test camera, native textures, and performance-critical paths on **physical devices**.
</Callout>

## GPUDeviceProvider

When multiple components need the same device (Three.js scene + overlay + debug UI), share one device with [`GPUDeviceProvider`](/api/gpu-device-provider):

```tsx
<GPUDeviceProvider deviceDescriptor={{ requiredFeatures: ["rnwebgpu/native-texture"] }}>
  <Scene />
  <DebugOverlay />
</GPUDeviceProvider>
```

Child components read the shared device via [`useMainDevice`](/api/use-main-device). Request features once at creation - you cannot add features later without a new device.

## Native texture feature

Camera and video import require the `"rnwebgpu/native-texture"` feature. It is **enabled by default** when the adapter supports it - you only need to check before importing:

```tsx
if (!device.features.has("rnwebgpu/native-texture" as GPUFeatureName)) {
  return; // rare on some Android emulators
}
```

For explicit requests at creation time:

```tsx
adapter.requestDevice({
  requiredFeatures: ["rnwebgpu/native-texture" as GPUFeatureName],
});
```

See [Native Textures](/docs/native-media/native-textures) for the full import guide.

## Threading and device.lost

WebGPU can run from the main JS thread, Reanimated UI thread, or dedicated worklet runtimes. However:

<Callout type="warn" title="device.lost and uncapturederror">
These events are only delivered on the **main JS runtime**. Create the device on the main thread and pass it to worklets when you need loss/error handling. If you create the device inside a worklet, loss events will not fire there.
</Callout>

See [Reanimated](/docs/integrations/reanimated) for UI-thread rendering patterns.
