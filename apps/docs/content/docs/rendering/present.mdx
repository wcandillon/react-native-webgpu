On the web, the browser swaps your rendered frame to the screen automatically when you submit GPU commands. On React Native, **you must call `context.present()`** after each `queue.submit()` to display the result.

This is the most important React Native–specific difference from browser WebGPU. Every tutorial that works in Chrome needs this one extra line on native.

<Callout type="warn" title="API may be removed">
`context.present()` is a React Native–specific extension today, but it may be removed in a future release - see [PR #357](https://github.com/wcandillon/react-native-webgpu/pull/357), which aligns presentation with the Skia-style model (automatic swap after submit). Until that lands, you still need to call `present()` after each `queue.submit()` on native.
</Callout>

## The frame loop

The minimum loop looks like this:

```tsx
const encoder = device.createCommandEncoder();
const pass = encoder.beginRenderPass({ /* color attachment */ });
// setPipeline, draw…
pass.end();

device.queue.submit([encoder.finish()]);
context.present(); // ← required on React Native
```

`present()` runs **synchronously** on the calling thread. That means you can present from:

- The main JS thread (`requestAnimationFrame`)
- The Reanimated UI thread (`runOnUI`)
- A dedicated worklet runtime or Vision Camera frame processor

Whichever thread encodes and submits must also call `present()`.

## Animation loop

Wrap the encode → submit → present sequence in a recurring callback:

```tsx
const render = () => {
  const textureView = context.getCurrentTexture().createView();
  const encoder = device.createCommandEncoder();
  const pass = encoder.beginRenderPass({
    colorAttachments: [{
      view: textureView,
      clearValue: [0, 0, 0, 1],
      loadOp: "clear",
      storeOp: "store",
    }],
  });
  pass.setPipeline(pipeline);
  pass.draw(3);
  pass.end();

  device.queue.submit([encoder.finish()]);
  context.present();
  requestAnimationFrame(render);
};
requestAnimationFrame(render);
```

Always call `present()` **after** `submit()`, never before. For native texture imports, the order is typically **submit → present → destroy external texture** - see [External Textures](/docs/native-media/external-textures).

## Async work between frames

GPU queue submission is asynchronous - the CPU returns immediately while the GPU works in parallel. That is usually fine, but **blocking the JavaScript event loop** with heavy async work between frames can starve `requestAnimationFrame` and make rendering stutter.

Guidelines:

- Keep per-frame setup synchronous where possible (update uniforms, encode passes, submit, present)
- Avoid `await`ing large downloads or model parsing inside the animation callback
- Load assets once at startup, then only touch GPU resources in the render loop
- For readback (`mapAsync`), prefer dedicated threads - see [Reanimated](/docs/integrations/reanimated)
